## Reproduction Steps

1. Fresh machine with Docker, no AWS credentials
2. `curl -fsSL https://raw.githubusercontent.com/reposwarm/reposwarm-cli/main/install.sh | sh`
3. `reposwarm new --local`
4. `ANTHROPIC_API_KEY=sk-ant-xxx reposwarm config provider setup --provider anthropic --non-interactive`
5. `reposwarm preflight` â†’ **FAIL: missing ANTHROPIC_API_KEY**
6. `reposwarm repos add is-odd --url https://github.com/jonschlinkert/is-odd` â†’ **500: MissingAuthenticationToken**

## Bugs Found

### Bug 1 (Critical): `ANTHROPIC_API_KEY` never written to `worker.env`

**File:** `internal/config/provider.go`, function `WorkerEnvVars()` (line ~155)

`WorkerEnvVars()` builds the env vars map for the worker container. It:
1. Iterates `b.EnvVars.Always` â€” only sets vars with a fixed `Value` field
2. For Anthropic, `ANTHROPIC_API_KEY` has `required: true` but no `value` (user-supplied) â†’ skipped
3. The `switch` statement handles `ProviderBedrock` and `ProviderLiteLLM` but has **no `case ProviderAnthropic`**
4. `ProviderConfig` struct has no field to store an Anthropic API key

**File:** `internal/commands/config_provider.go`, `newProviderSetupCmd()`

- No `--api-key` flag exists for anthropic
- Interactive wizard prompts for Bedrock (region/auth/keys) and LiteLLM (proxy URL/key) but **Anthropic falls through with zero prompts**
- Even if `ANTHROPIC_API_KEY` is set as an env var, `WorkerEnvVars()` never reads from `os.Environ()`

**Result:** After `config provider setup --provider anthropic`, `worker.env` contains `ANTHROPIC_MODEL` and `ANTHROPIC_SMALL_FAST_MODEL` but **never** `ANTHROPIC_API_KEY`.

### Bug 2 (Medium): Stale Bedrock env vars survive provider switch

**File:** `internal/commands/config_provider.go`, `writeWorkerEnv()` (line ~952)

`writeWorkerEnv()` **merges** new vars into existing `worker.env` â€” existing vars not in the new map are preserved. When switching from Bedrock to Anthropic:

- `WorkerEnvVars()` returns only `ANTHROPIC_MODEL` + `ANTHROPIC_SMALL_FAST_MODEL`
- Old `CLAUDE_CODE_USE_BEDROCK=1` and `CLAUDE_PROVIDER=bedrock` are never removed
- Worker sees Bedrock vars and tries AWS authentication â†’ fails

The unit test at `provider_test.go:85` validates `WorkerEnvVars()` doesn't *add* Bedrock vars for Anthropic, but doesn't test that `writeWorkerEnv()` *removes* stale vars.

### Bug 3 (Critical â€” API repo): DynamoDB Local fails on cred-less machines

**Repo:** `reposwarm/api` (not this CLI repo)

The docker-compose.yml generated by `reposwarm new --local` (`internal/bootstrap/local.go` ~line 701) doesn't include `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` for the API service. On machines without AWS credentials, DynamoDB Local operations fail with `MissingAuthenticationToken`.

The API server's DynamoDB client appears to use a credential provider that doesn't pick up env vars correctly â€” manual testing from inside the same container with the same env vars works fine via the default credential chain.

**Fix:** Add dummy AWS credentials to the API service in the generated docker-compose.yml template.

## Fixes Applied (this commit)

1. **Bug 1:** Added `APIKey` field to `ProviderConfig`, `--api-key` flag + interactive prompt for Anthropic, `case ProviderAnthropic` in `WorkerEnvVars()` that sources from config then falls back to `os.Getenv("ANTHROPIC_API_KEY")`
2. **Bug 2:** Added provider-aware cleanup in `writeWorkerEnv()` â€” removes known vars from other providers when switching
3. **Bug 3:** Added `AWS_ACCESS_KEY_ID=fakeAccessKey` and `AWS_SECRET_ACCESS_KEY=fakeSecretKey` defaults to the API service in the docker-compose.yml template
